# 奥数题库 MVP 产品需求文档

---

## 1. 项目概述

### 1.1 项目背景

- 小学奥数（数学思维训练）市场规模达 48 亿元，年复合增长率 12.3%
- 现有产品多为“大而全”的静态题库，缺乏“深入浅出”的动态讲解能力
- 家长和学生的核心痛点：**“题目有答案，但看不懂推导过程”**

### 1.2 产品定位

一款 **以高质量讲解为核心** 的轻量级奥数思维训练工具。不做大而全的题库堆砌，而是通过 **分步图文 + 多解法对照 + 可视化场景 + （后续）AI 引导** 的组合，让每一道题都“有人带着想清楚”。

### 1.3 目标用户

| 用户类型        | 核心诉求                                     |
| :-------------- | :------------------------------------------- |
| 3-6 年级小学生  | 有奥数兴趣或培优需求，需要清晰易懂的解题思路 |
| 陪读家长        | 希望有能力辅导孩子，但自身对奥数解法不熟悉   |
| 独立老师/小机构 | 需要高质量的讲解素材辅助教学                 |

---

### 1.4 词汇表

| 术语   | 解释                                     |
| :----- | :--------------------------------------- |
| MVP    | Minimum Viable Product，最小可行产品     |
| 变式题 | 与母题知识点相同、情境或数据不同的衍生题 |

## 2. MVP 范围

### 2.1 MVP 目标

跑通“价值感知 → 持续回访”的完整链路，验证用户是否认可“讲得清楚”这件事，并据此决定后续是否进一步投入内容生产与付费能力。

### 2.2 MVP 包含功能

| 模块       | 功能点                                  | 优先级 | 说明                              |
| :--------- | :-------------------------------------- | :----- | :-------------------------------- |
| 题库首页   | 按年级 / 模块浏览题目列表、全局搜索     | P0     | 入口页，支持随机一题等发现式入口  |
| 题目展示   | 题干文字 + 可选 SVG 图示                | P0     | 关键数据/条件清晰呈现             |
| 图文解析   | 多解法分步拆解 + 结构化可视化场景       | P0     | 所有解法完整展示，不做解锁墙      |
| 举一反三   | 结构化输入的变式练习 + 正误判断 + 提示  | P0     | 验证学习效果                      |
| 知识点讲解 | 每题下方的相关定理/方法讲解入口（占位） | P1     | MVP 以占位+“即将上线”收集意愿信号 |
| 关联推荐   | 同类题目推荐（related problems）        | P1     | 便于刷同一方法的变式              |
| 知识点标签 | 关联解题方法/模型标签                   | P1     | 与 `@/src/lib/tags.ts` 白名单对齐 |

### 2.3 MVP 不包含功能

- **视频讲解**（V2 再做，MVP 阶段重资产 ROI 过低）
- 用户登录/注册系统
- 任何付费 / 留资 / 解锁墙相关流程（MVP 完全免费开放）
- 题库后台管理系统（题目以 JSON 形式直接入库）
- AI 对话引导（V3 规划）
- 学习进度追踪与数据看板

## 3. 详细功能设计

### 3.1 题目信息模块

- **内容**：题干文字 + 关键数据高亮
- **示例**：
  > 笼子里有鸡和兔，共 **30个头**，**88只脚**。问鸡和兔各有多少只？

### 3.2 图文解析模块

同一道题提供多种解法分步拆解，全部默认可见。每个解法由若干 `steps`（文字）与可选的 `scenes`（结构化可视化片段，如比较条、头脚图、方程列表等）组成，做到“读文字 + 看图示”同步推进。

下面以“鸡兔同笼”为例展示三种解法的节奏：

**方法一：画图法**

- ① 画 30 个圆圈代表 30 个头

  ```
  ○ ○ ○ ○ ○ ○ ○ ○ ○ ○
  ○ ○ ○ ○ ○ ○ ○ ○ ○ ○
  ○ ○ ○ ○ ○ ○ ○ ○ ○ ○
  ```

- ② 每个下面画 2 只脚（假设全是鸡）→ 共 60 只脚

  ```
  ○ ○ ○ ○ ○ ○ ○ ○ ○ ○   ← 30 个头
  ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥   ← 每个 2 只脚
  ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥   合计：30 × 2 = 60 只脚
  ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥
  ```

- ③ 实际 88 只，还差 28 只

  ```
  目标  ████████████████████████████████████████████  88
  当前  ████████████████████████████████              60
                                        ↑ 还差 28 只脚
  ```

- ④ 每只鸡补 2 只脚变兔 → 补 14 只（28 ÷ 2 = 14）

  ```
  ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ │ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○ ○
  ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ │ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥ ∥
  ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ │
  ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ ╳ │
       🐰 兔（4 脚）× 14        │        🐔 鸡（2 脚）× 16
  ```

- ⑤ 结论：兔 14 只，鸡 16 只

  ```
  🐰×14 + 🐔×16 = 30 只（头）
  4×14 + 2×16 = 56 + 32 = 88 只（脚）✓
  ```

**方法二：假设法**

- ① 假设全是鸡：30 × 2 = 60 只脚
- ② 实际相差：88 - 60 = 28 只脚
- ③ 每只兔比鸡多 2 只脚
- ④ 兔数：28 ÷ 2 = 14 只
- ⑤ 鸡数：30 - 14 = 16 只

**方法三：方程法（高年级适用）**

- ① 设鸡 x 只，兔 y 只
- ② x + y = 30
- ③ 2x + 4y = 88
- ④ 解得：x = 16，y = 14

### 3.3 举一反三模块

- **变式题目**：
  > 停车场上共有汽车和摩托车 28 辆，轮子总数 86 个。汽车 4 轮，摩托车 2 轮，问各有多少辆？
- **交互**：结构化字段输入（单/多字段），提交后显示正误判断，错误时可展示 `hint`
- **可见性**：与主解法一致，默认完全开放

### 3.4 知识点标签模块

- 展示本题关联的**解题方法 / 模型标签**（取自 `@/src/lib/tags.ts` 的白名单，如 `鸟头模型` `假设法` `容斥原理`）
- 年级 / 模块不进 `tags`，它们已经是独立字段
- 点击标签跳转到同类题目列表（V2 实现）

### 3.5 相关知识点讲解入口

- **位置**：题目解析模块之后、练习模块之前
- **内容来源**：优先读取题目 JSON 中的可选字段 `knowledgePoints: [{ slug, name }]`；若未显式声明，则从 `tags` 中按白名单自动派生
- **当前形态（MVP 占位）**：每个知识点渲染为一个可展开条目，点开后显示“讲解内容正在整理中，敬请期待”的占位文案
- **目的**：
  - 给出“有人管我是否真的懂这个定理”的信号，强化讲解型产品定位
  - 作为轻量探针，收集哪些定理/方法最常被点开，用于排期正式讲解内容
- **演进**：V2 将 `slug` 对应到独立的知识点讲解页（直观场景 → 推导 → 2–3 道递进例题），点击占位条目时直接跳转

## 4. 技术方案

### 4.1 技术栈选型

| 层级 | 技术                                     | 说明                           |
| :--- | :--------------------------------------- | :----------------------------- |
| 前端 | PNPM + Shadcn + Vite v8 + TailwindCSS v4 | 灵活，可扩展性强               |
| 部署 | Vercel                                   | 免费，自动 CI/CD               |

### 4.3 页面结构

```
单题页面 (响应式 Web)
├── ProblemHeader：返回、题号、标题、年级/模块 Badge、方法标签
├── QuestionCard：题干 + 可选 SVG 图示
├── SolutionTabs：多解法图文解析（全部开放）
├── KnowledgePoints：相关知识点讲解入口（MVP 为占位）
├── PracticeCard：举一反三练习（结构化输入）
└── RelatedProblems：同类题目推荐
```

## 5. 验收标准

### 5.1 功能验收

- [ ] 题目信息正确展示（题干 + 可选图示）
- [ ] 多解法全部默认可见，步骤与可视化场景对应正确
- [ ] 变式题输入答案后可判断正误，必要时展示提示
- [ ] 知识点标签正确显示，且均来自方法标签白名单
- [ ] 相关知识点讲解入口正确渲染，占位条目可展开
- [ ] 页面在移动端与桌面端均可正常显示

### 5.2 体验验收

- [ ] 图文解析步骤分明，易于跟随
- [ ] 可视化场景与文字步骤语义一致
- [ ] 页面加载速度 < 2 秒

### 5.3 产品验证指标

| 指标             | 目标值 | 说明                                    |
| :--------------- | :----- | :-------------------------------------- |
| 解析读完率       | > 50%  | 滚动到解析底部 / 总访问                 |
| 多解法切换率     | > 30%  | 切换过至少一次解法 Tab / 总访问         |
| 知识点入口点开率 | > 15%  | 展开任一知识点条目 / 总访问（需求信号） |
| 变式题尝试率     | > 40%  | 提交过一次变式答案 / 总访问             |
| 关联题跳转率     | > 20%  | 点击关联题卡片 / 总访问                 |

## 6. 后续迭代规划

### V2（1-2 个月后）

- **正式知识点讲解页**：把 MVP 的占位入口替换为独立路由（如 `/knowledge/:slug`），含直观场景、推导与 2–3 道递进例题
- **引入视频讲解**（3 分钟手绘风格，基于前期验证后的需求强度决定是否自制 or 外包）
- 持续扩充题目与模块覆盖，知识点与题目做显式关联
- 按需再评估用户系统与付费能力

### V3（3-6 个月后）

- 引入 AI 对话引导（用户上传题目，AI 分步反问引导）
- 可视化知识图谱（展示题目间的关联）
- 多端适配（微信小程序版本）
